## P0. CFP / 登壇審査員の視点

### 指摘 1: フォーマット（LT）と内容スコープが過積載
**重大度**: 高
**該当箇所**: line 25「このLTでは、Generative UIプロトコルのコードリーディングから見えた「収束パターン」を整理し、AIがUIのユーザーになる世界で、フロントエンドはどんな境界を引くべきか——その問いを、一緒に考えます。」
**問題**: 「LT」と明記されている（通常5分前後）。にもかかわらず内容は ①複数プロトコルのコードリーディング → ②収束パターンの抽出 → ③フロントエンドの責務論、という3段構え。abstract に出てくる固有名詞だけでも MCP-UI / MCP Apps / A2UI / json-render / OpenUI / OpenAI と6つあり、1つ1分も割けない。「収束」を説得力をもって示すには各プロトコルの構造を最低限見せる必要があり、それだけで尺を食い尽くす。
**根拠**: 収束を主張するには「複数事例の内部構造」を提示する必要があり、これは構造上 LT の尺と両立しない。20〜40分のセッション相当のボリューム。
**提案**: (a) LT枠なら主張を1点に絞る。例:「json-render の型区分けを軸に、`catalog` という新しい境界面の話だけをする」。(b) フルセッション枠があるならそちらへ。**冒頭で枠（LT/セッション）を確定し、それに合わせて内容を削るか移すか**を決める。

### 指摘 2: 聴衆の持ち帰り(takeaway)と対象レベルが弱い
**重大度**: 中
**該当箇所**: 締めが line 25「その問いを、一緒に考えます」。
**問題**: 問題提起で閉じており「聴衆が何を持ち帰れるか」が明示されない。審査員は「面白い問いだが、聞いた人の明日が変わるか?」を見る。問いかけ型自体は良いが、LTでも「この視点を持つと自分の設計判断がこう変わる」という一撃が欲しい。
**提案**: abstract 末尾に1文、具体的なtakeawayを置く。例:「AIにUIを生成させる際、『何を許すか』を catalog 設計として書き下す観点を持ち帰れます」。

## P1. 関数型志向フロントエンドエンジニアの視点

### 指摘 1: 「収束パターン」が既に公開済みの分類と重複し、新規性が問われる
**重大度**: 高
**該当箇所**: line 19「どれも細かな手法は違えど、大きな枠は3-4層の境界に分離されていました。」/ line 25「コードリーディングから見えた「収束パターン」を整理し」
**問題**: Generative UI を「制御↔自由度」の軸で3類型に整理する公開記事が既にある。すなわち Static(AG-UI) / Declarative(A2UI・json-render系) / Open-ended(MCP Apps)。「3-4層に収束する」という主張がこの既存分類の言い換えに見えると、フロント中級層には「知ってる話の再整理」と受け取られる。
**根拠**: CopilotKit "The Developer's Guide to Generative UI in 2026" が上記3類型を制御-自由度スペクトラムで提示済み。
**提案**: 既存分類を**先に引用して土俵に乗せた上で**、自分の差分を明示する。例:「既存は『どこに制御を置くか』の分類。本トークは『各層の界面に現れる型の区分け(catalog/spec/primitive)』という直交軸で見る」と差別化する。

### 指摘 2: AG-UI への言及がなく、サーベイに穴がある
**重大度**: 中
**該当箇所**: line 9「様々なプロトコルが模索されており」（列挙に AG-UI が無い）
**問題**: 一番フロントエンドに近い「Static パターン」の代表 AG-UI（CopilotKit）が抜けている。フロント勢からは「フロントに最も近い規格を外して『収束』を語るのは?」と必ず突かれる。
**提案**: AG-UI を列挙に加えるか、「本トークは宣言的spec型に絞る、AG-UIのようなevent/state型は対象外」と明示的にスコープを切る。

### 指摘 3: トレードオフ比較が「共通項」に潰されている
**重大度**: 中
**該当箇所**: line 21「これらのライブラリにはAIをユーザーと見立てる上での共通項のような部分もあります」
**問題**: declarative(allowlistカタログ) と open-ended(iframeサンドボックス) は、表現力・安全性・FW非依存性で正反対のトレードオフを取る。それを「共通の制約」でまとめると、関数型勢が最も知りたい『使い分けの分岐点』が消える。
**提案**: 「共通項」と同じ熱量で「分岐点（どの制約を優先するとどの層設計に倒れるか）」を見せる。むしろそこが論の山場になる。

## P2. フレームワーク作法で書いているジュニアエンジニアの視点

### 指摘 1: 対象レベルが未明示で、冒頭から用語が洪水
**重大度**: 中
**該当箇所**: 冒頭〜line 19。catalog/spec/primitive/render、MCP、MCP-UI、MCP Apps、A2UI、json-render が短時間に列挙される。
**問題**: 「Generative UI」自体の平易な定義が無いまま固有名詞が走り出す。MCP を触ったことのない読者は2文目で振り落とされる。CFPとしては許容範囲でも、トーク本編がこの密度だとジュニアは置いていかれる。
**提案**: 1行でいいので接地する。例:「Generative UI＝AIが用意済みの部品を選んでUIを組み立てて返す仕組み」。あわせて対象レベル（"MCP/AIエージェントを触ったことがある人向け" 等）をabstractに明記。

### 指摘 2: catalog / spec / primitive の差が初見で掴めない
**重大度**: 低
**該当箇所**: line 13-16
**問題**: 4語を括弧内の一言ずつで導入しているが、初学者には spec(中間表現) と primitive(実体) の違いが直感的に区別できない。
**提案**: トーク本編では「catalog=メニュー表 / spec=注文票 / primitive=実際の料理」のような図解・喩えを必須にする。abstractでもこの3語のうち1つは具体例を添える。

## P4. 型システム・言語仕様に強いシニアエンジニアの視点

### 指摘 1: 「3-4層」という曖昧な層数、かつ render を層に数える分類の粗さ
**重大度**: 高
**該当箇所**: line 16「両者をつなぐ描画処理(render)」/ line 19「3-4層の境界に分離されていました」
**問題**: catalog/spec/primitive は『境界面（データ契約）』だが、render は『処理（プロセス）』。性質の異なるものを同じ「層」に並べ、しかも「3-4層」と数を曖昧にしているのは分類として粗い。「renderは層ではなく層間の射(変換)では?」と即座に突かれる。
**提案**: 「3つの界面(catalog/spec/primitive) ＋ それらを貫く描画処理(render)」と役割を明確に分離する。層を「契約層 / 中間表現層 / 実体層」と命名し、render は変換器として別立てにすると整合する。

### 指摘 2: 「OpenAI」「OpenUI」をプロトコルと並置する分類エラー
**重大度**: 高
**該当箇所**: line 19「A2UI、OpenUI、MCP Apps、OpenAIなど」
**問題**:
1. **"OpenAI" は企業名でプロトコル名ではない**。実体は OpenAI Apps SDK であり、それは MCP / MCP Apps の上に構築されている。MCP Apps と並置すると同じものを二重に数えることになる。
2. **"OpenUI" は曖昧で土俵違い**。wandb/OpenUI は自然言語→HTML/コードを生成するプロトタイピングツールであり、catalog/spec/primitive/render を持つランタイム規格ではない（さらに別物の thesysdev/OpenUI も存在し、名前が衝突している）。収束パターンの実例として並べると比較軸がずれる。
**根拠**: OpenAI Apps SDK は MCP/MCP Apps ベース（developers.openai.com/apps-sdk）。wandb/OpenUI の README は「describe UI → 描画 → HTMLをReact/Svelte等へ変換」と説明しており、ランタイムのUI契約規格ではない。
**提案**: 列挙を「同じ土俵（宣言的 Generative UI 規格）」に揃える。OpenAI は "OpenAI Apps SDK" と正名し **MCP Apps の実装例**として位置づける。OpenUI は列挙から外すか、「コード生成系は別カテゴリ」と注記して区別する。

### 指摘 3: MCP-UI と MCP Apps の関係（系譜）が未整理
**重大度**: 中
**該当箇所**: line 7（MCP-UI が昨年大手採用）↔ line 9（MCP Apps を「模索中」の規格として列挙）
**問題**: MCP Apps(SEP-1865) は MCP-UI のコミュニティ（MCP-UI Community Working Group）と共同で、MCP-UI のアプローチを標準化したもの。両者を「話題になった先行事例」と「模索中の規格」として無関係に並べると、標準化の経緯を知る聴衆には整理不足に映る。
**根拠**: MCP公式ブログ（2025-11-21 発表 / 2026-01-26 正式リリース）、SEP-1865。MCP-UI の採用企業（Postman/Shopify/Hugging Face/ElevenLabs/Goose）。
**提案**: 「MCP-UI（事実上の先行実装）→ MCP Apps（その標準化、2026-01 正式リリース）」と系譜を1行で示すと、聴衆の理解と論の信頼性が上がる。

### 指摘 4: 「デファクトが見えていない」の現状認識が2026年6月時点でやや古い可能性
**重大度**: 低
**該当箇所**: line 9「いまだデファクトスタンダードといえる規格は見えていない状態です」
**問題**: MCP Apps は2026-01に正式リリースし、ChatGPT/Claude/Goose/VSCode が対応済み。少なくとも『埋め込み型(open-ended)』ではデファクト化が進行している。"declarative spec 層に限れば未確定" は妥当だが、無条件の言い切りは最新性で突かれる。
**提案**: 「埋め込み型は MCP Apps に収束しつつある一方、宣言的spec型(A2UI/json-render)はまだ未確定」と層別に書くと正確。

## P5. アプリケーション / サプライチェーンセキュリティエンジニアの視点

### 指摘 1: 「AIは信頼できない」の脅威モデルが未定義で、catalog の保証範囲を超えた示唆になりうる
**重大度**: 高
**該当箇所**: line 21「「AIは信頼できない」」/ line 23「「AIに何を許すかを設計する」こともフロントエンドの責務」
**問題**: 「何から守るのか」が無い。catalog による allowlist は『使えるコンポーネントの種別』を縛るが、『prop に入る値』は縛らない。例えば Text に注入された誘導文言を出すフィッシング、Link/Action に悪性 URL を入れる遷移誘導は、catalog を通過してしまう。「カタログ＝境界面」だけでは安全の証明にならず、ここを曖昧にすると「allowlistすれば安全」という過大主張に滑る。
**根拠**: json-render 公式も catalog を "guardrail" と表現するに留め、安全保証とは言っていない。MCP Apps が iframe サンドボックスを必須化したのは、宣言的 allowlist だけでは防げない領域があることの裏返し。
**提案**: 「catalog が縛れるもの(型/コンポーネント種別) と縛れないもの(値/文言/URL/副作用)」を切り分け、後者にはサニタイズ・URLスキーム制限・実行前確認UI等の**別レイヤ**が要ると明示する。これは弱点ではなく**トークの核**にできる論点。

### 指摘 2: 三類型でセキュリティのトレードオフが正反対なのに「共通制約」で平滑化している
**重大度**: 中
**該当箇所**: line 21「「AIは信頼できない」「実装を知らせたくない」「逐次生成に耐える」「フレームワーク非依存にしたい」という共通の制約」
**問題**: declarative(allowlist) は表現力を捨てて安全側に倒し、open-ended(iframe) は表現力を取りサンドボックスで隔離する。守り方の思想が逆。これを「逐次生成に耐える」「FW非依存」と同列に並べると、安全の議論が単なる設計選好の議論に薄まる。
**提案**: 「どの脅威を受容/隔離するかで層の置き方が決まる」と、セキュリティを**分岐の主因**として立てる。共通制約ではなく分岐点として扱う。

### 指摘 3: 「実装を知らせたくない」をセキュリティ制約とするなら根拠が弱い
**重大度**: 低
**該当箇所**: line 21「「実装を知らせたくない」」
**問題**: これは情報隠蔽/結合度の話であり、obscurity を安全の根拠にすると "security by obscurity" と取られかねない。意図が『AIと実装の結合を切る抽象化』なら、その言葉で語るべき。
**提案**: 「実装詳細を晒さない＝AIへの結合を断つ抽象化のため」と意図を明示し、安全性の根拠とは切り離す。

## 追記: 5分LT・構成「1-2-2」前提での再評価（2026-06-21）

著者から補足あり。前提を更新する。

- **枠は5分のLT**（フルセッションではない）。
- **構成は 1-2-2**: ①1分で「Generative UIとは何か」、②2分で json-render / A2UI / MCP-UI の処理の流れ、③2分で「共通項」＋「AIをユーザーと見立てる際のフロントエンド境界はどうあるべきか」という著者の持論。
- 天気UI等の実装デモは20分版の構想で、5分では断念（妥当な判断）。

この前提で、初回レビューの各指摘を再評価する。

### 取り下げ・降格する指摘（＋自分の誤りの訂正）

- **【訂正】P2-1「Generative UIの平易な定義が無い」は誤り**。line 5 に「AIエージェントが規定のコンポーネント群からUIを組み立て、ユーザー(人間)に返却を行う」という定義が既にある。指摘を撤回する。さらに著者は本編冒頭1分で定義する計画であり、トーク側でも手当て済み。残る有効部分は「line 9 の固有名詞の洪水」のみ（重大度: 低に降格）。
- **P1-2「AG-UIへの言及が無い」→ 取り下げ寄り（低）**。5分で3つに絞るのは適切なスコープ。ただし abstract が line 9 で「様々なプロトコルが模索されており」と広く構えている点だけは、実際に扱う3つと齟齬がある（後述の abstract 書き直しで解消）。
- **P0-1「内容が過積載」→ 論点シフト（中）**。著者が 1-2-2 に意識的にスコープ済みなので「内容を削れ」は解消。残る問題は「**abstract がその 1-2-2 構成を反映しておらず、6規格のサーベイに読めてしまう**」こと。対処は内容削減ではなく abstract の書き直し。
- **P1-3 / P5-2「トレードオフを共通項で平滑化」→ Q&A準備として低**。5分では「共通項→持論」に絞るのが妥当。差分(使い分け)はQ&Aで聞かれ得るので口頭で答えられるよう準備しておけば十分。

### 5分LT前提でも残る／むしろ強まる指摘

- **P4-2「OpenAI / OpenUI をプロトコルと並置」→ 強まる（高）**。トーク本編で扱うのは json-render / A2UI / MCP-UI の3つだけ。OpenUI・OpenAI・MCP Apps は本編に登場しない。**登場しない上に分類エラーを抱えている**のだから、abstract から落とすのが最善。これで abstract と本編が一致する。
- **P4-1「3-4層の曖昧さ・render を層に数える粗さ」→ 残る（高）**。フォーマットに依存しない分類の問題。本編の「2分で処理の流れ」を語る際の骨格そのものなので、ここを「3界面＋描画処理」と整理しておくことがトークの説得力に直結する。
- **P5-1「『AIは信頼できない』の脅威モデル未定義」→ 残る（高）**。最後の2分の持論＝まさに境界論。catalog だけで安全を主張しない（値・URL・文言は別レイヤ）という線引きを、持論の中で一言入れるだけで過大主張を回避でき、むしろ主張が締まる。

### 【新規・最重要】3例の「収束」は宣言的カタログ族に限った話で、MCP-UI は remote-dom モードでしか成立しない
**重大度**: 高
**該当箇所**: 著者の構成②「json-render / A2UI / MCP-UI の処理の流れ」＋ line 19「3-4層の境界に分離」
**問題**: 選んだ3例のうち、json-render と A2UI は「宣言的カタログ型」で catalog→spec→primitive→render の骨格に素直に乗る。しかし **MCP-UI はモードによって構造が違う**:

| MCP-UI のモード | 構造 | 「収束」骨格に乗るか |
|---|---|---|
| `rawHtml` | サーバが HTML 文字列を返し iframe で描画 | ✗ catalog も spec/primitive 分離も無い |
| `externalUrl` | 外部アプリURLを iframe で読み込む | ✗ 同上 |
| `remote-dom` | サーバが UI構築スクリプトを送り、変更をJSONで host に伝え、**host のコンポーネントライブラリ**で描画 | ✓ host components＝primitive、送られるツリー＝spec として乗る |

mcpui.dev の入門例は `rawHtml`。それを見せて「json-render/A2UI と同じ骨格に収束」と言うと、構造が違うので破綻する。P4 から「収束に都合のいい remote-dom モードだけをチェリーピックしている」と突かれる。
**根拠**: MCP-UI 公式ドキュメント（resource-renderer / getting-started）。remote-dom は Shopify の remote-dom を用い、host 提供のコンポーネントライブラリで描画する（＝primitive 層に対応）。
**提案**:
1. **MCP-UI は remote-dom モードで見せる**と明言する。これは弱点ではなく、Shopify remote-dom の host-provided components がそのまま「primitive 層」に対応するため、**持論を補強する好例**になる。
2. さらに踏み込むなら「収束しているのは“宣言的カタログ型”の一族（json-render / A2UI / MCP-UI:remote-dom / AG-UI）であって、“埋め込み型”（MCP-UI:rawHtml / MCP Apps の iframe）は別の収束（サンドボックス済みドキュメントを丸ごと渡す）に向かう」と**2つの収束**として整理すると、主張がより正確かつ強くなる。境界論(P5-1)にもきれいに接続する。

### 【新規・演出】時間制約を武器に変える：3規格を「同一の骨格図」で見せる
**重大度**: 中（提案）
**問題/機会**: 2分で3規格＝1規格あたり40秒。個別に語ると確実に薄くなる。
**提案**: catalog→spec→primitive→render の**同一スケルトン図**を出し、名前のラベルだけを json-render / A2UI / MCP-UI(remote-dom) で差し替えて3回見せる。視聴者は「箱は同じで中身の名前だけ違う」と一目で理解でき、**「収束」という主張そのものが視覚的に証明される**。40秒×3の制約が、逆に収束を伝えるレトリックになる。

### 最大の打ち手: abstract を本編（1-2-2）の形に書き直す

上記を総合すると、最も費用対効果が高いのは **abstract を実際のトーク構成に一致させる書き直し**。下記はドラフト（proposal.md 本体はまだ未編集。適用可否は著者判断）。

> **title案（原案を維持しつつ短縮も可）**
> AIがユーザーになる時代、フロントエンドはどこに境界を引くか ― Generative UI に共通する設計
>
> **description案**
> AIエージェントが用意済みの部品を選んでUIを組み立て、人間に返す「Generative UI」が注目されています。本LT（5分）は3部構成です。(1) Generative UI とは何かを最初に押さえ、(2) json-render・A2UI・MCP-UI(remote-dom) という3つの実装が、いずれも「カタログ（AIが読む語彙）→ 中間表現 → コンポーネント実体 → 描画」という同じ骨格に収束していることを、共通のスケルトン図で見せます。(3) そこから「AIをユーザーと見立てるとき、フロントエンドはどこに境界を引くべきか」という私の考えにつなぎます。これまで相手は人間の目に映るHTML/CSSでした。これからは“AIに何を許すか（＝カタログ）”が新しい境界面になる――この一点を持ち帰ってもらうトークです。

- 扱わない OpenUI / OpenAI / MCP Apps(単体) を列挙から外し、本編3規格に一致させた（P4-2解消）。
