## 1. 決定事項（このFBで決めたこと）

- **聴衆の大半は Generative UI を知らない**前提に立つ。先に定義しないと「自分の設計思想のすごさ」が伝わらない（比較対象が無いと"すごさ"が測れない）。
- 5分は「広く横断」を諦め、**前提（定義）→ 流れ → 主張** の一本道にする。
- 扱うプロトコルは **json-render / A2UI / MCP-UI の3つに限定**。OpenUI / OpenAI Apps SDK / UCP は概要から外す（本編で触れるなら正名・系譜つきで。→ §6 のP4指摘）。
- **明言するスコープ**: 本トークは「宣言的 spec 型（json-render / A2UI）＋ 埋め込み型（MCP-UI）」に絞る。**event/state 型（AG-UI）は対象外**と1枚で明言する（P1指摘2 の穴を、扱わないと宣言して塞ぐ）。
- 「天気カードを具体例で貫く」案（旧・第2案）は **20分版に温存**。5分ではサンプルの流れ説明まで確実に間に合わない（→ §7 棲み分け）。

## 3. ②「流れ」パートの見せ方（最重要の尺対策）

**3本別々に説明しない。1本通し＋差分。**

1. **共通枠を1枚**: `AIが読む契約(catalog) → AIが書く宣言データ(spec) → 描画処理(render) → 人が書く実体(primitive)`。これを"型"として先に置く。
2. **json-render で1本通す（〜60s）**: `defineCatalog` で契約 → AIが JSONL patch で spec を吐く → `Renderer` が `registry[type]` で実体を描く。**「AIはコンポーネントを書かず、組み立て指示(spec)だけ書く」**を1回ちゃんと見せる。
3. **A2UI を差分（〜30s）**: 同じ枠。違いは **構造(updateComponents)とデータ(updateDataModel)が別メッセージ**な点だけ指す。
4. **MCP-UI を差分（〜30s）**: 同じ枠。違いは **実体がサーバ返却の iframe / rawHtml / RemoteDom** な点。ここで **「MCP-UI（先行実装）→ MCP Apps（2026-01 標準化, SEP-1865）」の系譜を1行**添える（P4指摘3）。

→ こうすると「3つの違う物を見た」でなく「**1つの枠に3つが乗った**」になり、③の「同じ形」が回収される。

---

## 4. ③ 共通項＋持論（主張の核）

1. **同じ形に収斂する**: 4つの制約「AIは信頼できない／実装を知らせたくない／逐次生成に耐える／FW非依存」が同じだから、別組織でも同じ層構造に行き着く。
   - ⚠️ **比喩は「収斂進化」**を使う。「コンウェイの法則」は誤用（組織構造→システムの話で方向が逆）。
2. **既存分類を土俵に乗せて差別化**（新規性の確保 / P1指摘1）: 「Generative UI を _制御↔自由度_ で3類型に分ける見方は既にある（CopilotKit 2026）。本トークはそれと直交する **"各界面に現れる型の区分け（catalog/spec/primitive）"** という軸で見る」と最初に断る。これで"まとめ記事"化を防ぐ。
   - ⚠️ 層の言い方は **「3つの界面(catalog/spec/primitive) ＋ それらを貫く描画処理(render)」**。render は層でなく変換器（P4指摘1）。「3-4層」と数を濁さない。
3. **持論（自分にしか言えない核）**: **catalog は"型/コンポーネント種別"は縛れるが、"値・文言・URL・副作用"は縛れない**（P5指摘1）。だから「AIに何を許すか」を考えると、UI生成(AIに任せる)と**データ権威(人/システムが握る)を分離**する必要がある——コミュニティサービスでAIがホームUIを組む機能を設計したとき、自分は **データを spec に焼かず外部権威に READ時委譲する（ContentSource）** 判断をした。これが「AIをユーザーと見立てる時の境界」の自分の答え。
4. **takeaway 一撃**（P0指摘2）: 「AIにUIを作らせるとき、_何を許すか_ を `catalog` として書き下す——その視点を持ち帰ってほしい」。

## 5. 改訂プロポーザル（第4案 / `proposal.md` の置き換え候補）

### title（70字以内）

▎ **Generative UIって何？ — AIがUIのユーザーになる時代の、フロントエンドの新しい境界**

- 代替（横断を強調する版）: ▎ AIがUIを組み立てる時代のフロント境界 — json-render / A2UI / MCP-UI に共通する設計を5分で

### 概要（約450字 / 1000字以内）

> 「AIに『今日の天気は？』と聞くと、文章じゃなくて"天気カード"が返ってくる」——こんなふうに、AIがその場でUIを組み立てて返す仕組みを **Generative UI** と呼びます。まだ耳慣れない分野ですが、ここ1年で json-render(Vercel)、A2UI(Google)、MCP-UI など各社のプロトコルが一気に出てきました。
>
> このLTでは、まず「Generative UIとは何か」を1分で掴んでもらい（前提知識は要りません）、次に json-render / A2UI / MCP-UI が"どう動くか"の流れをさっと見せます。すると、手法は違うのに、どれも同じ形に行き着いていることが見えてきます——AIが読む契約(catalog)、AIが書く宣言的データ(spec)、人が書く実体(primitive)、それを描く処理(render)。
>
> なぜ同じ形に収斂するのか。そして、ユーザーが人間ではなくAIになったとき、フロントエンドはどこに境界を引くべきなのか。コミュニティサービスでAIがUIを組む機能を設計した経験から、私なりの答えを最後にお話しします。AIにUIを作らせるとき「何を許すか」を設計する——その視点を持ち帰ってもらえたら、と思います。
